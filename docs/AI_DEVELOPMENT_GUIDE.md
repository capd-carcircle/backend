# AI 코딩 에이전트와 함께 개발하며 정리한 원칙

이 프로젝트의 상당 부분은 AI 코딩 에이전트와 대화하며 진행했다. 도구를 오래 쓰다 보니 몇 가지 반복되는 실패 패턴과, 그것을 피하기 위해 자리 잡은 습관이 생겼다. 아래는 "AI에게 코드를 맡길 때 실제로 도움이 됐던 것들"을 기록한 것이다. AI 자체를 다루는 요령이라기보다, AI를 매개로 일할 때 결국 사람이 챙겨야 하는 부분에 관한 기록에 가깝다.

## 1. 작업을 작은 단위로 쪼개고, 매 단위마다 승인받는다

큰 변경을 한 번에 맡기지 않고, "이번엔 이 범위만" 하는 식으로 작업을 슬라이스로 나눴다. 예를 들어 다른 언어 스택으로 로직을 이관할 때는 먼저 순수 계산 로직만 포팅해서 정합성을 증명하는 슬라이스를 완료하고, 그다음 세션에 인증·캐시 같은 나머지를 붙이는 슬라이스로 넘어갔다. 슬라이스를 나누면 한 세션에서 검증까지 끝낼 수 있는 크기로 작업이 줄어들고, 다음 슬라이스로 넘어갈지 다른 우선순위로 갈지를 매번 사람이 판단할 수 있는 지점이 생긴다.

계획에 없던 "하는 김에 해볼 만한" 작업(보너스 스코프)과 원래 계획했던 필수 스코프는 항상 구분해서, 보너스 쪽은 우선순위와 리스크를 먼저 설명하고 착수 여부를 확인받았다.

## 2. 로직을 복제하거나 이관할 때는 정합성 검증부터 만든다

같은 계산 로직이 여러 곳에 존재하게 되는 상황(다른 서비스로 포팅, 다른 언어로 이관 등)에서는 기능을 다 만들고 나서 검증하는 게 아니라, 검증 방법부터 먼저 정했다.

- **같은 언어끼리 비교**할 때는 시드를 고정한 난수 생성기를 테스트 코드가 공유하게 만들어서, "같은 입력엔 항상 같은 기대값"이 성립하도록 했다.
- **다른 언어끼리 비교**할 때는 난수 생성 알고리즘 자체가 달라 시드 공유가 통하지 않으므로, 한쪽 언어의 실제 출력을 고정된 fixture 파일로 저장해 두고 다른 쪽이 그 fixture와 일치하는지 비교하는 방식으로 바꿨다.
- 반올림 방식, 부동소수점 문자열 변환처럼 평소엔 신경 쓰지 않는 지점이 크로스랭귀지 이관에서는 실제 불일치의 원인이 됐다. 문제가 발견되면 언어별 표준 동작을 그대로 쓰기보다, 원본 언어의 동작을 명시적으로 재현하는 유틸리티를 따로 만들어 해결했다.

로직을 한 곳 더 복제해야 하는 상황이 생겼을 때, 공유 패키지로 분리할지 그냥 중복을 감수할지는 매번 트레이드오프로 판단했다. 레포/배포 파이프라인이 하나 더 늘어나는 비용이 몇백 줄의 중복보다 클 때는 중복을 택하고, 대신 자동화된 정합성 테스트로 어긋남을 감지하는 쪽을 택했다.

## 3. 실행 환경의 파일 읽기 결과를 맹신하지 않는다

AI 에이전트가 코드를 실행하는 샌드박스 환경과, 사람이 실제로 파일을 저장하는 위치가 마운트 방식으로 연결되어 있는 경우, 방금 수정한 파일을 다시 읽었을 때 이전 버전이나 깨진 내용으로 보이는 현상을 여러 차례 겪었다. 이 상태에서 자동 검증(문법 체크, 테스트 실행)을 돌리면 실제로는 멀쩡한 코드가 오류로 보고된다.

이런 상황에서 실제로 도움이 된 원칙은 "가장 낮은 레벨의 파일 읽기 결과를 최종 진실로 삼는다"는 것이었다. 자동화된 검사 결과가 이상하면 우선 의심하고, 별도 경로로 파일을 다시 읽어 실제 내용을 재확인한 뒤 판단했다. 반대로 사람이 로컬 환경에서 직접 실행한 검증 결과(빌드 로그, CI 결과)는 항상 신뢰할 수 있는 근거로 취급했다.

## 4. 자동화가 실행한 형상관리 명령의 결과도 같은 이유로 의심한다

같은 마운트 신뢰성 문제는 소스 코드뿐 아니라 버전 관리 상태 확인에도 영향을 줬다 — 이미 반영된 파일이 마치 아직 수정된 것처럼 보이거나, 원격 저장소의 실제 이력과 다른 상태가 보고되는 경우가 있었다. 이 때문에 다음 두 가지 원칙을 정했다.

- 자동화가 실행한 상태 확인 명령의 결과만으로 "반영 완료"를 판단하지 않고, 실제로 사람이 실행한 명령의 출력(성공/실패, 커밋 해시)만을 최종 확인 수단으로 삼는다.
- 커밋·푸시처럼 되돌리기 까다로운 작업은 자동화가 대신 실행하지 않고, 사람이 직접 실행하도록 역할을 분리한다. 권한이 애매한 상태에서 자동화가 형상관리 명령을 실행하면, 잠금 파일이 정리되지 않고 남는 등 예상치 못한 부작용이 생기기 쉽다.

## 5. 캐시 무효화 로직은 실제 데이터 변경 경로를 코드로 확인한 뒤 설계한다

캐시를 도입할 때 "혹시 모르니 자주 무효화한다"는 방어적인 접근 대신, 캐시된 데이터가 실제로 바뀔 수 있는 코드 경로가 있는지부터 확인했다. 예를 들어 특정 상태로 넘어간 데이터는 이후 어떤 API로도 그 안의 수치를 바꿀 방법이 코드상 없다는 것을 직접 추적해 확인한 뒤에는, 신선도 체크 없이 존재 여부만으로 캐시를 재사용하도록 단순화했다. 나중에 그 가정이 깨지는 기능(예: 이미 제출된 데이터를 수정할 수 있게 되는 변경)이 추가되면 그때 무효화 로직을 다시 넣어야 한다는 점을 문서에 명시해 뒀다 — 지금 당장 필요 없는 방어 로직을 미리 넣지 않는 대신, "언제 이 가정이 깨지는지"를 남겨두는 방식이다.

## 6. 실패를 전체 작업의 실패로 만들지 않는다

여러 건을 순회하며 처리하는 배치성 작업에서는 항목 하나의 실패가 전체를 멈추지 않도록 개별적으로 예외를 처리했다. 마찬가지로, 부가적인 기능(예: 실패 알림 전송)이 설정되어 있지 않다고 해서 핵심 작업 자체가 실패로 처리되지 않도록 구분했다. 무엇이 핵심이고 무엇이 부가 기능인지 구분해서 실패 처리 정책을 다르게 가져가는 것이 배치 작업 설계에서 중요했다.

## 7. 작업 단위마다 배경-문제-해결-검증 순서로 기록한다

각 변경을 커밋 메시지 한 줄로 남기는 것과 별개로, 별도 작업 로그에는 항상 같은 순서로 기록했다: 왜 이 작업이 필요했는지(배경), 무엇이 문제였는지, 어떻게 해결했는지, 그리고 무엇을 어떻게 검증했는지(그리고 아직 검증하지 못한 부분은 무엇인지). 이 형식을 지키면 다음에 이어받는 사람(또는 시간이 지난 뒤의 자기 자신)이 "왜 이렇게 했는지"를 코드만 보고는 알 수 없는 부분까지 복원할 수 있다. 특히 검증하지 못한 부분을 숨기지 않고 명시해 두는 것이 중요했다 — AI 에이전트가 실행 환경의 제약(권한 없음, 네트워크 제한 등)으로 끝까지 확인하지 못한 부분은 반드시 남겨서, 사람이 그 지점만 콕 집어 확인할 수 있게 했다.

## 요약

이 원칙들을 관통하는 하나의 태도는, AI 에이전트가 빠르게 만들어내는 결과물을 그대로 신뢰하지 않고 "무엇을 어떻게 검증했는가"를 항상 별도로 확보한다는 것이다. 코드를 빨리 작성하는 것보다, 그 코드가 맞다는 것을 무엇으로 증명할 수 있는지를 함께 만들어 두는 쪽이 결국 더 빠르다.
